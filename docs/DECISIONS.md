세션별 진행 문서에 흩어져 있던 **"왜 이렇게 결정했는가"**를 한곳에 모은 문서다.

- **어떻게 만들어졌는가** (구조·좌표계·물리 모델·상태머신·점수룰) → [GAME_DESIGN.md](GAME_DESIGN.md)
- **왜 이렇게 결정했는가** (밸런스·튜닝·연출·UI 방향) → 이 문서
- **무엇을 언제 했는가** (세션 로그·커밋) → [archive/](archive/) (PROGRESS·GAMEPLAY_ROADMAP·SPIN_FEEL_AND_AI_LADDER·UI_REVAMP·POLISH_BACKLOG)

> 물리 수치는 튜닝으로 바뀔 수 있으니 **코드(`src/game/constants.ts`)가 최종 진실**이고, 여기 적힌 값은 "그 결정을 내릴 때의 근거"로 읽을 것. 각 항목은 **결정 → 근거** 순.

## 1. 게임 방향 · 범위

### 1.1 확정된 방향

- **상대는 필요** — AI 라이벌 확정. 단 리텐션(사람 모으기)은 목표 아님 → 데일리 시드 등은 후순위 고정.
- **세션 길이** — 10프레임 풀게임이 볼링의 정체성이므로 기본 유지. 경량 모드(3프레임 블리츠 등)는 대체가 아니라 **메뉴 선택지**로.
- **코어 밸런스 문제 확인** — 직구가 훅보다 강함(실측, §2). 훅이 보상받게 만드는 게 최우선 디자인 과제.
- **모바일 지원** — 터치 발사·반응형 UI·safe-area·햅틱. 터치 스핀은 전용 UI 신설 대신 **스핀 바 드래그**로 해결(Q/E 병행). 상세는 [MOBILE_SUPPORT.md](MOBILE_SUPPORT.md).
- **지향점**: **"볼링을 가르치는 세미시뮬"로 잠정 확정.** 아케이드 캐주얼(빠른 쾌감) vs 세미시뮬(라인 읽기·오일 메타 숙련)을 두고, 코드베이스가 이미 세미시뮬로 투표했다 — Rule of 31 브레이크포인트·훅 드리프트 캘리·예측선 3단·레인 마름·성격 AI 사다리. 손맛(§3)은 숏/롱 강도 *튜닝*엔 필요하나 방향 결정엔 더는 블로커가 아니다. (상세: [OIL_META_AND_AUTO.md](OIL_META_AND_AUTO.md) §2.7.)

### 1.2 개선 아이디어 검토 결과

여러 개선 아이디어를 검토하고 채택/보류를 결정한 기록.

| 아이디어 | 결정 | 근거 |
|---|---|---|
| 스핀 튜닝(손맛) | **채택** | sim 검증 끝, 실플레이 체감 미확정이라 최우선(§3) |
| AI 난이도 차별화 | **채택** | 3인이 난이도가 아니라 스타일로만 분화돼 실력 거의 동급이었음(§6) |
| 승리 시 보상 | 보류 → 이후 채택 | 당시 보상 정의 미정. 이후 업적→코스메틱으로 구현([REWARDS.md](REWARDS.md)) |
| 에셋 보정 | 보류 | 실에셋 교체가 아니라 **절차적 다듬기 방향**으로, 나중 |
| 핀 충돌 시각효과(플래시/스파크) | 백로그 | 비용 대비 체감 크다고 봤으나 우선순위에서 밀림 |

### 1.3 우선순위 순서

**손맛 → 코어 밸런스 → 온라인성/보여주기 → AI 상대 → 연출 → 예측선 정책 → 리텐션.**
손맛 없이는 방향 결정 근거가 없고, 코어 밸런스 없이는 훅(핵심 자산)이 장식이고, AI가 동기를 만들고, 연출이 체감을 가장 크게 바꾸고, 예측선은 잘 만든 물리의 보람을 회수한다. 리텐션은 무관심 결정에 따라 하향 — 스페어 드릴·볼 컬렉션 정도만 생존 후보.

## 2. 코어 밸런스 — "직구 250" 문제 (훅이 보상받아야 한다)

**결정**: 직구 천장을 ~180-200으로 낮추고, **200+는 훅으로만** 열리게 밸런스한다.

**근거**:

**플레이어 실측**: 실제 볼링에서 직구 점수가 180인 플레이어가, 이 게임에서는 **직구만으로 ~250 달성**. 게임의 직구가 현실보다 훨씬 후하다는 직접 증거.

**시뮬 실측** ([sim-carry.mjs](../sim-carry.mjs) — 핀 10개 풀 시뮬, 진입 x를 1cm 간격 31회 스캔):

| 구질 | 평균 진입각 | 스트라이크 |
|---|---|---|
| **직구 풀파워** | 0.2° | **8/31** ← 당시 최적해 |
| 풀스핀 훅 풀파워 | 4.1° | 7/31 |
| 풀스핀 훅 미드파워 | 3.8° | 4/31 |
| 직구 미드파워 | 0.2° | 3/31 |

**원인 — 두 겹의 문제**:
1. **진입각이 캐리에 보상을 안 줌** (속도가 전부) — 실볼링과 반대(진입각 4~6°가 200+의 열쇠).
2. **실행 노이즈 부재** — 직구 윈도우 ±8cm는 실볼링이면 몸의 분산 때문에 못 때리지만, 마우스 조준은 노이즈 0이라 매번 적중 → 직구 천장이 250까지 뚫림. 실볼링의 직구 천장(~180)은 물리가 아니라 실행 분산이 만드는 것.

**레버**:
- **① 핀 캐리**: 직진 진입은 디플렉션으로 코너 핀이 남게, 진입각은 캐리 보상 — 핀 restitution(↓), 핀-핀 에너지 전달, 볼-핀 질량비(무거운 공 우대), 핀덱 마찰. → §5에서 실행.
- **② 실행 텐션**: 릴리스 타이밍 — 게이지 정점 이탈 시 aim 노이즈. 노이즈 0인 마우스 입력에 분산을 되돌려줌. 레버 ①만으로 직구 천장이 안 잡히면 승격.

**측정 가능한 목표**: 직구 풀파워 윈도우 8/31 → 3~4/31로 축소 + 훅 풀파워 ≥ 직구의 2배. (레버 ② 효과 측정은 노이즈 0 윈도우만 재는 현재 도구로 불가 — "aim 노이즈 주입 기대 점수" 모드 추가 필요.)

⚠️ 손맛(§3)과 묶어 진행 — 캐리를 바꾸면 체감 점수대가 통째로 움직인다.

## 3. 스핀 손맛 — `spin^0.7` 입력 곡선

**결정**: 물리 레버가 아니라 **스핀 입력을 `spin^0.7`로 리매핑**한다 (`constants.ts` `SPIN_POW=0.7` / `effectiveSpin()`, [Ball.ts](../src/scene/Ball.ts) 발사·[Controls.ts](../src/input/Controls.ts) 예측선 공용). 물리 보정이 아닌 아케이드성 입력 곡선.

**근거**:

**왜 밋밋한가** — [Ball.applySpinForce](../src/scene/Ball.ts:127):
```ts
const fMag = FRICTION_K * REF_MASS * 9.81 * hook;   // ← 스핀 양과 무관한 상수
```
- 측면력 **세기는 스핀 양과 무관** — 스핀 양은 *슬립 방향*과 *지속 시간*에만 작용.
- 훅은 드라이 존(`z > OIL_END_Z=10.5`)에서만 `hookFactor`로 켜짐.
- **풀파워**: 빠르니 슬립이 드라이 존까지 살아 들어옴 → 막판 강한 스냅 = "확 휜다".
- **미드파워·약스핀**: 드라이 존 도달 전 슬립이 닫히거나 횡성분이 작음 → 밋밋.

**왜 물리 레버가 아니라 입력 리매핑인가** — 물리 레버(`ROLL_RATIO`/`SLIP_EPS`/`FRICTION_K`/`OIL_END_Z`/`HOOK_RAMP`)를 그리드로 전수 스캔한 결과 **전부 저/미드스핀 dead zone을 못 살리고 가드만 붕괴**시켰다. 훅은 횡슬립 비율이 스핀에 비례하므로, 어떤 물리 레버도 풀스핀 가드를 안 깨고 약스핀을 못 살린다. 측면력 *크기*를 스핀에 비례시키면 오히려 `hook ∝ 스핀²`이 되어 약스핀이 *더* 죽는다.

→ 해법은 스핀 *입력*을 `spin^p`로 리매핑. **`1.0^p = 1.0`이 고정점**이라 풀스핀과 모든 가드(−30cm · 4/31 · 7/31 · 65cm)가 **자동으로 불변**이고, 저역(약스핀)만 끌어올린다.

**sim 검증** (`--spinPow 0.7`): 저스핀 막판 스냅 −2.8 → **−4.1cm(+40%)**, 미드 −5.2 → −6.6, 풀스핀·윈도우·총휨은 베이스라인과 동일. (0.5는 +100%지만 저스핀 총휨 과대 → 0.7 채택.)

> ⚠️ 총 변위와 체감 스냅은 다르다. sim상 풀스핀 미드파워(−61cm)가 풀파워(−33cm)보다 총휨이 큰데도 미드가 밋밋하게 느껴지는 건, **막판 곡률(스냅)·가독성** 부족으로 "휨"이 아니라 "흘러간 misaim"처럼 읽히기 때문. 목표는 총 변위 증가가 아니라 **"의도된 늦은 스냅"의 가독성 확대**.

**가독성 레버 (물리와 별개)** — 예측 조준선은 전체 곡선(`PREVIEW_N=32`)을 계산하고도 `setDrawRange(0, 8)`로 앞 8점(짧은 직선)만 그린다([Controls.ts:83](../src/input/Controls.ts)). "훅 결과 숨김"(조준 스킬 대체 방지) 설계라, 던지기 전 어디로 휠지가 안 보여 밋밋함을 키운다. → UI 개편(§8)에서 다룸.

## 4. 마찰 결합 규칙 — 레인 `Min` / 핀 `Max`

**결정**: 레인 바닥 콜라이더는 `setFrictionCombineRule(Min)`, 핀 콜라이더는 `setFrictionCombineRule(Max)`. 바닥 단일 콜라이더를 공유하면서 접촉 쌍별 마찰 정책을 분리한다.

**근거**:

**레인 Min** — Rapier 마찰 결합 기본값이 Average라 레인 오일(0.04)이 공 마찰(0.1)과 평균돼 오일 존 μ가 0.05 밑으로 못 내려간다 → 슬립(훅 연료)이 오일 존에서 일찍 닫혀 훅의 절반이 오일 존에서 새고 막판 스냅이 안 난다. 바닥에 `Min` 결합 + `LANE_FRICTION_OIL` 0.04→0.015, `FRICTION_K` 0.12→0.16, `SPIN_RATE` 9→14.
- 결과(풀스핀 미드파워 10lb): 오일 끝 −8cm(직진 스키드) → 헤드핀 −61cm, **훅의 87%가 드라이 존** = 스키드→스냅. 풀파워는 −33cm(빠르면 덜 휨, 현실적). 무게 역전도 해소(6lb −63 vs 16lb −66).

**핀 Max** (레인 Min의 부작용 교정) — 레인 Min 결합이 핀-레인 마찰까지 오일값으로 끌어내려(0.22→0.14) 핀이 토플 대신 슬라이드했다("리셋 안 된 듯" 보임). 핀 콜라이더에 `Max` 결합을 주면 Rapier 규칙 우선순위(**Max > Min**)로 핀-레인은 항상 0.3 고정, 공-레인 오일 시뮬은 무영향.
- 결과: 센터 풀파워 8핀 + 생존 핀 스팟 이탈 0.8cm/0cm(수정 전 수십 cm). 핀-공 마찰도 0.2→0.3으로 올라 토플이 더 잘 됨.

## 5. 핀 캐리 밸런스 — 선형 감쇠 0.8 / 반발 0.3

**결정**: `PIN_LINEAR_DAMPING=0.8` + `PIN_RESTITUTION=0.3` (`constants.ts`). ([sim-carry.mjs](../sim-carry.mjs)를 `--pinRest --pinDamp --pinMass --pinFric --ballRest --pinComY`로 파라미터화해 그리드 스캔.)

**근거**:
- 직구 풀파워 윈도우 8/31 → **4/31**, 훅 풀파워 7/31 유지 = 직구의 1.75배(목표 2배에 1카운트 부족).
- **핀 선형 감쇠가 "날아가는 핀의 운 좋은 체인"을 죽이는 게 직구만 선택적으로 깎는 유일한 레버**였다. 단일 레버(반발↓ / 마찰↑ / 질량↑ / COM↓)는 전부 훅을 *더* 깎아서 못 씀.
- 부족분(2배까지 1카운트)은 릴리스 타이밍 레버(§2 레버 ②)가 후보.

## 6. AI 난이도 사다리 — 난이도=분산 축 / 개성=스타일 축

**결정**: AI 라이벌 3인을 **난이도(분산)**와 **개성(스타일)** 두 직교 축으로 설계한다. 상대를 고르면 난이도가 따라온다(메뉴 구조 그대로).
- **난이도(분산)**: `aimJitterCm`, `powerJitter`, `spareAimJitterCm`. 작을수록 강함.
- **개성(스타일)**: 직구/훅/스페어형 정체성 유지.

**근거 — 스핀(§3)을 먼저 확정한 뒤 AI를 캘리해야 하는 순서 의존성**: AI 훅형의 조준 캘리브레이션(`HOOK_DRIFT_HOUSE` — [ai.ts](../src/game/ai.ts):80 `hookDriftFor()`의 하우스 앵커)이 스핀 물리에서 유도된 값이다. 스핀 레버(`SPIN_RATE`/`FRICTION_K`/`ROLL_RATIO`)를 바꾸면 이 드리프트가 달라져 AI 조준이 어긋난다. → **물리 확정 후 그 위에서 AI 점수대를 재측정**해야 캘리를 두 번 안 한다.

**매치 sim이 잡아낸 캘리브레이션 버그**: AI 직구가 `POCKET_X_STRAIGHT=0`(헤드핀 정면=노즈히트=스플릿)을 노려 거의 스트라이크가 안 났다 — 모든 프로필이 jitter 무관 ~120-156에 뭉친 진짜 원인. 미세스윕으로 실제 직구 포켓 = entry **−7cm** 확인 → `POCKET_X_STRAIGHT` 0→**−0.07** 수정 후 사다리가 벌어졌다. 훅 포켓도 `POCKET_X_HOOK` 0.067→**0.05** 재조정.

**구현 제약 (코드 대조 확인)**:
- ⓐ `.mjs`는 `.ts`를 import 못 한다(`tsx`/`ts-node` 미설치) → 매치 sim은 **vitest 기반 `.ts` 스크립트**로 짜야 `Scoreboard.totalScore`/`computeAiThrow`(둘 다 순수함수)를 그대로 쓴다([gameplay.test.ts](../tests/gameplay.test.ts) 선례).
- ⓑ `GameState`의 10프레임 흐름은 재사용 불가 — Three/Rapier/DOM과 얽혀 헤드리스 import 불가. 투구 생성·프레임 진행·10프레임 보너스 흐름을 ~80-150줄 신규 작성해야 한다(스코어링 식 자체는 `Scoreboard` 재사용).
- ⓒ 스페어 분기는 투구별 Rapier 핀 시뮬 필수 — `computeAiThrow` 스페어 경로가 남은 핀의 실제 x를 먹으므로, 투구마다 핀을 실제로 굴려야 `spareAimJitterCm` 차별이 산다. 통계적 근사로 단순화하면 스페어·10프레임에서 계통오차.

## 7. 카메라 충격 연출 폐기

**결정**: 충돌 시 카메라 셰이크/FOV 펀치를 **전면 폐기**. 충격은 **핀 물리(fly-out)·사운드**로 표현한다.

## 8. 인게임 UI — 하단 도크 통합

**결정** (레퍼런스 조사 후 확정):
1. **레이아웃 → 하단 도크 통합** — 스핀(가로) + 파워를 하단 한 묶음으로, 같은 패널 스타일·액센트 규칙으로 통일. 스핀은 가로 유지(좌우=훅 방향 직관성 보존), 가운데를 비워 공·조준선 노출.
2. **파워 존 → 흐리게 암시만** — 상단~중상(0.6~0.9) 은은한 골드 띠 + 진입 경계선만, 정밀 조준은 실력에 맡김.
3. **조준선 끝점 마커 → 짧은 끝점만** — `Z_CAP` 근처 방향/진행만 표시, 훅 최종 포켓 진입 결과는 계속 숨김(§3 "앞부분만" 철학 유지).

**근거 — 진단한 4가지 문제**:
- **① 파워 게이지가 "정보 0"** — 빈 캡슐로 읽힘. 라벨·눈금·휴지 채움이 없어 처음 보면 파워인지 모름.
- **② 스핀↔파워 입력 비대칭** — 같은 투구 입력인데 스핀은 하단 풀폭 가로 바, 파워는 우측 짧은 세로 바. 형태·위치·크기·액센트 색이 다 흩어져 한 쌍으로 안 읽힘.
- **③ 중립(흰색) 조준선이 밝은 레인에 묻힘** — 스핀 0에서 흰 선이 레인색(tan)에 녹음. 가장 자주 쓰는 중립이 가독성 최악.
- **④ 공이 스핀 패널에 가림** — 풀폭 스핀 패널 + 하단 중앙 렌더 공이 같은 영역을 점유.

**차용 레퍼런스**:
| 레퍼런스 | 차용 |
|---|---|
| PGA TOUR 2K25 EvoSwing | 우측 세로 파워 미터의 **채움 + 타깃 존 마커 + 눈금** → 파워 게이지에 스트라이크 최적 파워 존 |
| 8 Ball Pool 가이드라인 | **고대비 궤적선 + 끝점 접촉 인디케이터** → 중립색 고대비화 + 끝점 마커 |
| Bowling King / My Bowling 3D | 가볍고 투명한 컨트롤 UI로 레인·공 안 가림 → 패널 슬림·투명화 |

**교훈**:
- **핑퐁 차징에 라인 길이/모양을 묶으면 무조건 펌프질** → 입력 가이드는 파워와 분리.

## 9. 점수판(Hud) 레이아웃 — 한 줄 꽉 채우기

**결정**: 방향 A(한 줄에 10프레임 꽉 채우기). 2행(5+5)과 가로 스크롤은 **둘 다 기각**.
- **솔로 / 멀티 활성**: 행에 정해진 폭 `row { width:min(96vw, NAT_SHEET(+NAME_W)); max-width:96vw }`을 주고, 프레임 `flex:{2|3} 1 0`·셀 `flex:1 1 0`(basis 0)로 비례 분배. 여유 있으면 자연폭, 좁으면 균일 축소. 2칸/마지막 3칸 = grow 2:3이라 모든 셀 폭 동일. "한 줄 10프레임"(중계 그래픽) 관례 유지 = 2행 기각 사유 해소.
- **멀티 비활성(AI)**: `renderMini` — `F{프레임}` + 현재 프레임 마크 + 큰 총점. 폭·높이 절약, 상대는 총점이 핵심. **게임 종료 시엔 전원 풀 시트**로 최종 비교.
- **유지**: 보너스 확정 전 누적 숨김(`frameScores` undefined 게이팅), 표준 마크 X/–, 현재 프레임 골드 펄스.

> ⚠️ `fit-content`는 금물 — 초반 빈 칸을 0으로 산정해 collapse(실기 확인). §8 교훈과 동일.

## 10. 검증된 불변식 — 손대지 말 것 (재조사 방지)

리뷰에서 "잘 돼 있음"으로 확인된 것들. 건드리면 손해.

- **코어 루프/엔진** ([Loop.ts](../src/core/Loop.ts)·[Engine.ts](../src/core/Engine.ts)) — 고정 timestep accumulator + 보간, 벡터 재사용(hot path 무할당), timeScale/pause 인프라. 성능 clean.
- **CameraRig·Replay** — 모듈 레벨 스크래치 벡터/쿼터니언 재사용, Float32Array 스냅샷. 무할당.
- **점수 엔진·상태머신** — 300/275/블리츠/10프레임 보너스, `maxConsecutiveStrikes`(프레임 경계), 스플릿(슬리퍼 2-8/3-9 비스플릿·7-10), settle 타임아웃 vs 핀 안착 레이스 — 전부 재현 테스트로 정확.
- **타입 규율** — `as any`/`: any`/`@ts-ignore`/`debugger`/`TODO` 0건. 디버그 글로벌은 타입 선언으로 노출.
- **미사용 의심 export 4개** — `pinIndexByNumber`/`OIL_DRY_PER_FRAME`/`OIL_DRY_MAX`/`maxConsecutiveStrikes` 전부 live(내부+테스트 사용). **지우지 말 것.**

**코드 불변식 근거** (왜 이렇게 되어 있어야 하는가):

- **조준선 캐시 키에 볼 물성을 포함해야 함** — `speed`([Controls.ts:503](../src/input/Controls.ts))가 `ball.speedScale`, `inject`([Controls.ts:509](../src/input/Controls.ts))가 `ball.massKg`에 의존하고 `setSpec()`은 "AIMING 중에만 권장"([Ball.ts:100](../src/scene/Ball.ts))이라 **조준 중 무게 교체가 정상 경로**. `aim`/`spin`만 키로 쓰면 무게만 바꿨을 때 이전 볼 물리로 그린 조준선이 스테일로 남는다. (물성 대신 스펙 버전 카운터도 가능.) 캐시 적중은 잦음 — `aim`은 이산 대입, `spin`은 0.1 양자화 대입이라 매 프레임 스무딩이 없다. `resolution.set`은 리사이즈 반영이 필요하니 캐시 가드 밖에 두거나 리사이즈에서 무효화.
- **CSS는 성질별 3층 분리** — 정적/애니메이션만 진짜 `.css`로 빼고, 토큰은 TS 단일소스 유지(**WebGL이 읽음**), 런타임 계산 인라인은 TS에 둔다. 인라인 스타일 전체를 `.css`로 옮기는 건 이 구조(캔버스 + 명령형 DOM 오버레이, 프레임워크 없음)와 안 맞는다.
- **index.html 부팅 로더 CSS는 그대로 둔다** — 모듈보다 먼저 로드돼야 "즉시 타이핑" 연출이 되는 의도된 배치.
- **핀 프로파일은 `constants`에 둔다** (Pin.ts 아님) — Boot 순환 import 회피.

## 11. 전광판 — 비텍스트 "무드 스크린" (스틸컷과 역할 분리)

**결정** (레퍼런스 조사 후 확정): 핀 뒤 전광판([Environment.ts](../src/scene/Environment.ts))은 **점수·문구를 띄우지 않고**, 게임 상태를 **비텍스트 "무드"로만** 반영하는 반응형 스크린으로 둔다. "무엇이 일어났나"(STRIKE!/SPARE!/GUTTER)는 **스틸컷·HUD**(텍스트)가 전담. **스플릿 '경고' 배너는 제거**.

**근거 — 실제 볼링장의 2-surface 분리**:
- 실제 볼링장은 화면이 둘로 나뉜다 — ① **오버헤드 모니터**(레인 위, 볼러 정면) = 기능적 점수표(이름·프레임·X/–·누적), ② **마스킹 유닛 스크린**(핀 뒤/위) = 분위기·**축하 연출**(스트라이크 시 폭발·춤추는 핀 등). 게임의 전광판은 정확히 ②의 자리(핀 뒤 마스킹 월)라, 원래 "점수판"이 아니라 분위기/축하 스크린이 맞다.
- **스틸컷이 텍스트에 유리**: HUD 바로 밑·정면·큼직 → 필독 문구는 여기가 가독성 최고. 원경·비스듬한 전광판에 문구를 두면 손해. 그래서 **텍스트=스틸컷, 무드=전광판**으로 채널을 나눠 서로 안 겹치게 하고, 스트라이크는 문구(스틸컷)+환경 플레어(전광판) 둘로 커진다.
- **스플릿 제거**: 무드 스크린은 *좋은* 순간에 반응하는 게 자연스럽다. 스플릿은 다운 이벤트라 실제 축하 스크린도 안 띄운다 → 빨간 경고 배너가 컨셉과 어긋남.

**동작** (`pulse(kind, streak)` / `setActivePlayerTint(hex)`, 상수 `MOOD`([constants.ts](../src/game/constants.ts))):
- **energy** — 스트라이크 연속으로 상승(double→turkey→bagger 열기↑), 멈추면 자연 냉각. 태양 반경·그리드 속도·밝기를 몬다("온파이어").
- **flash** — 스트라이크(골드 블룸 + 밖으로 퍼지는 충격파 링) / 스페어(작은 버스트). 가산합성 "쾅".
- **dip** — 거터 = 순간 브라운아웃 + 플리커.
- **tint** — 현재 플레이어 네온 색([GameState.accentHex()](../src/game/GameState.ts))으로 은은한 크로스페이드(컬렉션 좌석색/신스랭크와 연동). 멀티에서만 발화, 솔로는 하우스 시안.
- **splitConverted → 스페어 펄스로 흡수** (스플릿 클리어 = 사실상 최고의 스페어, 관중 눈엔 그냥 깔끔한 스페어).

**차용 레퍼런스**:
| 레퍼런스 | 차용 |
|---|---|
| Brunswick Center Stage · QubicaAMF Harmony MasQ/Neoverse (LED 마스킹) | "매 스트라이크·스페어를 다이내믹 비주얼로 축하" → 비텍스트 무드 플레어 |
| 볼링장 "strike screen" 축하 애니(폭발·춤추는 핀) | 문구 대신 **환경 반응**(블룸·충격파·온파이어)으로 번안 |

> 관련: 스틸컷([StillCut.ts](../src/ui/StillCut.ts)) 텍스트는 전부 유지(직관성). 스플릿 감지([splits.ts](../src/game/splits.ts))는 화면 표시에서만 빠졌을 뿐 `splitConverted` 판정에 계속 쓰인다 — 지우지 말 것. 전광판은 여전히 **텍스트-프리**(브랜딩 마퀴만 유지).
