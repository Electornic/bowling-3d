# 게임성 로드맵 — 브레인스토밍 기록

> 작성: 2026-06-12 (4차 세션). 물리 코어 완성 후 "게임성을 올리려면 뭐가 필요한가" 논의 결과.
> v2: critic 리뷰 반영 — 난이도 재추정, reset 체크리스트 보완, P0 복원, 모바일 스핀 입력 격상.
> 현재 상태: 물리(훅·오일 패턴·핀 액션)는 탄탄, 게임 루프와 연출이 미완.

---

## 방향 결정 필요 (착수 전 확정)

- [ ] **지향점**: 아케이드 캐주얼(빠른 쾌감, Wii 볼링 톤) vs 세미 시뮬(라인 읽기·오일 메타 숙련)
  → P2 연출 톤과 P3 예측선 정책이 여기서 갈림. **P0 손맛 확인을 해봐야 정할 수 있는 결정** — 순서 주의.
- [ ] **모바일 지원 여부**: 현재 터치엔 스핀 입력 수단이 없음(Q/E 키보드뿐, `Controls.ts`). 모바일 데모 의도가 있으면 "터치 스핀 UI 신설"이 P1급으로 올라옴.

---

## P0. 손맛 확인 — 모든 것의 선행 (PROGRESS.md "다음 첫 할 일"과 동일)

4차 세션에서 스핀 물리를 크게 바꿨다(레이트 훅, Min/Max 마찰 결합). 실플레이 검증 없이는 지향점 결정도, P3 예측선 정책도 근거가 없다.

- [ ] 풀스핀 -61cm가 과한지 (과하면 `SPIN_RATE` 14→12)
- [ ] 핀 둔감이 Max 결합으로 해소됐는지 (부족하면 핀 무게중심 하향, restitution 조정)
- [ ] 레인 좁아 보임 → 팔로우캠 높이 1.5→1.2 등 카메라 먼저, 안 되면 `LANE_WIDTH` 치트(1.15~1.2)

## P1. 게임 루프 닫기 — 필수, 현재 "게임이 끝나지 않음"

설계 도안의 `Menu.ts`/MENU 상태가 미구현. 10프레임 후 GAME_OVER 상태에서 멈출 뿐 재시작 불가(새로고침 필요). 코드엔 MENU/PLAYING 상태 자체가 없고 곧장 AIMING으로 시작.

| 항목 | 구현 메모 | 난이도 |
|---|---|---|
| 재시작 — 최소안 | GAME_OVER 시 DOM 오버레이(최종 점수 + 재시작 버튼)만. `GameState.reset()` 신설 — **체크리스트: `frame=1`, `ball=1`, `rolls=[[]]`, `standingAtThrow=10`, `state='AIMING'`, `pins.resetAll()`, `ballObj.reset()`, `refreshHud()`** | 하 |
| 시작 화면 — 풀안 | `GameStateName`에 `MENU` 추가 → Controls(`=== 'AIMING'` 가드, Controls.ts), CameraRig(상태 switch), BallPicker(AIMING 가드)의 상태 분기를 전부 갱신하는 리팩터. BallPicker를 MENU 안으로 넣을지 상시 유지할지 UX 결정 필요 | 중 |
| 결과 화면 | 프레임별 점수 리캡 (`Scoreboard.frameScores` 재사용 — Hud가 import하는 그 함수) + 최종 점수 강조 | 하 |
| 하이스코어 + 통계 | localStorage: 최고점·평균·스트라이크%·스페어%. 스트라이크/스페어 판정은 rolls에서 프레임 분류 함수 신설 필요 | 하 |
| 스플릿 감지·표시 | `standingMask()`(PinSet.ts)는 있으나 **핀 인접 테이블이 없음** → 인접 그래프 신설 + "헤드핀 다운 & 잔여 핀 비인접" 판정 → HUD "7-10 스플릿!" | 하 |

## P2. 타격감 (juice) — 체감 게임성의 절반

볼링의 페이오프 = 임팩트 순간. 현재 건조함.

| 항목 | 구현 메모 | 난이도 |
|---|---|---|
| 스트라이크 슬로모 | ⚠️ **물리 dt는 절대 스케일하지 말 것** (결정성·궤적 검증 깨짐). `Loop`에 timeScale 추가하되 **accumulator 유입 시간만** 스케일(`acc += frame * scale`) — 각 스텝은 1/60 유지, 기존 렌더 보간이 끊김 흡수. **임팩트 트리거 신설 필요**: "첫 공-핀 충돌" 감지(ROLLING 중 ball.z > HEADPIN_Z−0.3 진입 시점이 가장 쌈) | 중 |
| 연속 스트라이크 피드백 | 더블/터키/포베거 텍스트 — rolls에서 연속 카운트만, 점수 로직 이미 있음. 싸고 임팩트 큼 | 하 |
| 핀덱 클로즈업 컷 + 텍스트 팝 | CameraRig 상태 추가 + "STRIKE!/SPARE!" DOM 팝 + 네온 플래시 | 중 |
| 카메라 셰이크 | 임팩트 contact force 크기 비례, 짧게 | 하 |
| 사운드 — 굴림 럼블 | 현 SoundManager는 `playHit` 단일 합성음뿐. 럼블은 노이즈+필터 지속 루프 + 매 프레임 공 속도 피드 (연결 신설) | 중 |
| 사운드 — 크래시 구분 | contact 이벤트는 force 크기만 줌 → 짧은 윈도우로 충돌 클러스터 카운트해 1~2핀 딸각 vs 풀랙 크래시 구분 | 중 |
| 사운드 — 스트라이크 환호 | **합성으론 불가, 실음원 필요** → P4 에셋과 묶기. GameState에 strike/spare 이벤트 노출이 선행 (P2 전반의 공통 의존) | 중 |
| 핀세터 스윕바 | 데드우드 제거가 현재 순간이동 → 스윕바가 쓸어가는 애니로 가림. 물리 없음, 시각 전용 | 중 |

## P3. 숙련 깊이 — 물리를 게임플레이로 환전

공통 선행: `hookFactor`/`OIL_END_Z`가 모듈 전역 상수이고 **Lane·Ball·Controls 3곳이 직접 참조** → 런타임 변경하려면 파라미터화(오일 상태 객체) 선행. 이거 하나 하면 아래 둘 다 싸짐.

| 항목 | 구현 메모 | 난이도 |
|---|---|---|
| 예측선 난이도 | 현재 예측선은 오차 ~1cm 오라클이라 조준 스킬을 대체해버림. 이지=풀라인 / 노멀=오일 존까지 / 프로=레인 화살표만. **지향점 결정(상단) 종속** | 하 |
| 오일 패턴 프리셋 | 하우스샷/숏/롱. 상수는 준비됐으나 위 파라미터화 필요 — "공짜" 아님 | 하~중 |
| 레인 전환(오일 마름) | 프레임 경과로 드라이 존 전방 확장. 파라미터화 선행 시 "하". 예측선도 같은 오일 상태를 봐야 정합 유지 | 하~중 |
| 릴리스 타이밍 | 파워 게이지 핑퐁 정점 근처 릴리스=정확, 벗어나면 aim 노이즈. 실행 스킬 추가 | 하 |
| 터치 스핀 입력 | 모바일 결정(상단) 시: 릴리스 직전 좌우 플릭 or 스핀 슬라이더. 설계 도안 §8의 플릭 안 참조 | 중 |

## P4. 콘텐츠/리텐션 — 나중

- 볼 컬렉션: 훅 포텐셜·컨트롤 다른 공 언락 (BallSpec 확장)
- 스페어 드릴 챌린지: 특정 핀 배치 픽업 (7-10 스플릿 등) — PinSet에 임의 배치 함수
- 로컬 2인 교대전: ⚠️ "GameState 2개"가 아님 — 분리할 건 **점수 상태(`frame/ball/rolls`)뿐**, 물리 객체(PinSet/Ball)는 공유 + 턴 전환 시 리셋. Hud 2인 점수판 대응 (중)
- 데일리 시드 챌린지 (시드 고정 오일 패턴)
- 비주얼/사운드 에셋: GLTF 핀·공, HDRI, 실음원(환호 등 P2 의존분) — 게임성보단 프레젠테이션, 마지막
- 범위 밖 백로그(하이스코어 외 통계·접근성·그래픽 옵션)는 GAME_DESIGN.md §12 P2 섹션 참조
- 파울 라인 판정(도안 §7의 'F')은 1인 게임 임팩트 낮아 **의도적 범위 밖** — 필요해지면 발사 위치 개념부터 신설

## 추천 순서

**P0 → P1 → P2 → P3 → P4.** P0 없이는 방향 결정 근거가 없고, P1 없이는 남에게 보여줄 수 없고, P2가 체감을 가장 크게 바꾸고, P3는 잘 만든 물리의 보람을 회수. P4는 위가 끝난 뒤.
(모바일 지원을 결정하면 "터치 스핀 입력"이 P1급으로 승격되는 것만 예외.)
