## P0.5. 코어 밸런스 — 훅이 보상받아야 한다 ("직구 250" 문제)

**플레이어 실측**: 실제 볼링에서 직구 점수가 180인 플레이어가, **이 게임에서는 직구만으로 ~250 달성**. 게임의 직구가 현실보다 훨씬 후하다는 직접 증거.

시뮬 실측 (2026-06-12, 핀 10개 풀 시뮬 [sim-carry.mjs](../sim-carry.mjs) — 진입 x를 1cm 간격 31회 스캔):

| 구질 | 평균 진입각 | 스트라이크 |
|---|---|---|
| **직구 풀파워** | 0.2° | **8/31** ← 현재 최적해 |
| 풀스핀 훅 풀파워 | 4.1° | 7/31 |
| 풀스핀 훅 미드파워 | 3.8° | 4/31 |
| 직구 미드파워 | 0.2° | 3/31 |

원인 분석 — 두 겹의 문제:
1. **진입각이 캐리에 보상을 안 줌** (속도가 전부) — 실볼링과 반대(진입각 4~6°가 200+의 열쇠).
2. **실행 노이즈 부재** — 직구 윈도우 ±8cm는 실볼링이면 몸의 분산 때문에 못 때리지만, 마우스 조준은 노이즈 0이라 매번 적중 → 직구 천장이 250까지 뚫림. 실볼링의 직구 천장(~180)은 물리가 아니라 실행 분산이 만드는 것.

**✅ 구현 완료 (5차 세션)**: 그리드 스캔 결과 핀 **선형 감쇠 0.8**(날아가는 핀 감속 → 운 좋은 체인 스트라이크 억제)이 직구만 선택적으로 깎는 유일한 레버였고, 핀 반발 0.2→**0.3**이 미드파워 캐리를 복원. 결과: 직구 풀파워 **8/31→4/31**, 직구 미드 2, 훅 미드 2, 훅 풀파워 **7/31 유지** (직구의 1.75배 — 2배 목표에 1카운트 부족하나 인접 조합 전부 열세. 부족분은 레버 ② 릴리스 타이밍이 후보). sim-carry.mjs는 CLI 오버라이드(`--pinRest --pinDamp --pinMass --pinFric --ballRest --pinComY`) 지원, 기본값은 게임과 동기.

- **목표**: 직구 천장 ~180-200, **200+는 훅으로만** 열리게
- **레버 ① 핀 캐리**: 직진 진입은 디플렉션으로 코너 핀이 남게, 진입각은 캐리 보상 — 핀 restitution(0.2↓), 핀-핀 에너지 전달, 볼-핀 질량비(무거운 공 우대), 핀덱 마찰
- **레버 ② 실행 텐션**: 릴리스 타이밍 — 게이지 정점 이탈 시 aim 노이즈. 노이즈 0인 마우스 입력에 분산을 되돌려줌. **현재는 P3 항목**(중복 아님) — 레버 ①만으로 직구 천장이 안 잡히면 P0.5로 승격 결정
- **측정**: sim-carry.mjs 재사용, 튜닝마다 4구질 윈도우 재측정. **측정 가능한 목표**: 직구 풀파워 윈도우 8/31 → 3~4/31로 축소 + 훅 풀파워 ≥ 직구의 2배. 레버 ② 효과 측정은 현재 도구로 불가(노이즈 0 윈도우만 잼) — "aim 노이즈 주입 기대 점수" 모드를 스크립트에 추가 필요
- ⚠️ P0(손맛)과 묶어 진행 — 캐리를 바꾸면 체감 점수대가 통째로 움직임

## P1.5. 상대 — AI 라이벌 — ✅ 구현 완료 (5차 세션)

구현: `ai.ts` 3인(김부장 안정 직구 / 한프로 스페어 장인 / 도박사 윤 풀스핀 도박형) — `(aim, power, spin)` 평균·분산 + sim 캘리브레이션(직구 진입 x≈aim×19.29, 훅 드리프트 0.33m). 점수 상태 플레이어별 분리, AI 턴 입력 락(Controls 가드), AI 턴 빨리감기(`Loop.timeScale` 3배 — accumulator 유입만 스케일), Hud 2인 시트 + 턴 배너. 스페어 처리는 잔여 핀 centroid 직구(와이드 스플릿은 자연히 어려움).

하이스코어만으론 동기가 약함. 성격 있는 AI 라이벌 2~3명과 매치 플레이 — "10프레임 굴리기"가 "김부장 꺾기"로 바뀜.

- 구현: 우리 물리를 그대로 쓰되 `(aim, power, spin)`에 성격별 평균·분산을 줘서 던지게 함 — 난이도·개성이 파라미터 몇 개 (스페어 정확형 / 풀스핀 도박형 / 안정 직구형)
- **선행 작업**: GameState 점수 상태(`frame/ball/rolls`)를 플레이어별 분리 — P4 로컬 2인전의 분리 작업과 동일한 것. 물리 객체(PinSet/Ball)는 공유. AI 턴 동안 사람 입력(Controls) 락도 필요
- 교대 투구 연출(상대 턴은 빨리감기 옵션) + 상대 점수판 — Hud 2인 대응도 P4 로컬 2인전과 공유
- 난이도: 중 (점수 상태 분리 + 게임 흐름 교대 + Hud 확장)

## P2. 타격감 (juice) — 체감 게임성의 절반 (✅ 6차 본편 완료, ⑥ 시각효과만 잔여)

볼링의 페이오프 = 임팩트 순간.
**5차 세션 선반영**: GameState strike/spare/split 이벤트 노출(P2 공통 의존 해소), `Loop.timeScale` 인프라(슬로모는 트리거·복원 정책만 남음), 연속 스트라이크 텍스트(STRIKE!/DOUBLE!/TURKEY!/N BAGGER!) + 스플릿/변환 배너(`Hud.banner`).
**✅ 6차 세션 완료**: 스트라이크 슬로모 트리거·복원(`GameState.onContact`→`Loop.timeScale`, `SLOWMO_SCALE`/`SLOWMO_REAL_SEC`), 임팩트 사운드 재설계(투구당 1회 크래시 — 서 있던 핀 수로 세기), 핀 접근 카메라(`CAM_APPROACH_Z`) + 임팩트 push-in(`PUSHIN_DIST` 0.6), 핀 리스팟(`respot` — 순간이동 제거), 전광판 일원화. 아래 표는 브레인스토밍 원본(상태는 이 줄 기준).
> **잔여(P2)**: **⑥ 충돌 지점 시각효과(플래시/스파크/닿은 핀 발광)** — `engine.onContact`에 force·위치 있어 배선 깔끔, 비용 대비 체감 큼(다음 배치 후보). · 카메라 셰이크는 **의도적 OFF**(`SHAKE_ENABLED=false`, push-in으로 대체). · 스트라이크 환호음은 P4 실음원 의존. · 핀세터 스윕바 애니는 미구현(리스팟으로 순간이동만 해소).

| 항목 | 구현 메모 | 난이도 |
|---|---|---|
| 스트라이크 슬로모 | ⚠️ **물리 dt는 절대 스케일하지 말 것** (결정성·궤적 검증 깨짐). `Loop`에 timeScale 추가하되 **accumulator 유입 시간만** 스케일(`acc += frame * scale`) — 각 스텝은 1/60 유지, 기존 렌더 보간이 끊김 흡수. **임팩트 트리거**: ROLLING 중 ball.z > HEADPIN_Z−0.3 근접 판정(프리롤용) 또는 기존 `engine.onContact` 실제 접촉 이벤트. **복원 정책 필수**: 실시간 ~0.8s 후 timeScale→1 — 안 돌리면 SETTLE_TIMEOUT(시뮬 4초)이 0.3배속에서 실시간 13초가 됨. 카메라 스무딩(onFrame)은 실시간 dt라 슬로모와 디싱크 — 연출 확인 필요 | 중 |
| 연속 스트라이크 피드백 | 더블/터키/포베거 텍스트 — rolls에서 연속 카운트만, 점수 로직 이미 있음. 싸고 임팩트 큼 | 하 |
| 포켓/훅 학습 피드백 | 스트라이크 시 "포켓 히트!" + 진입각 표시, 첫 판에 "Q/E 스핀으로 포켓을 노려보세요" 힌트 1회 — **훅→포켓→스트라이크 루프를 가르침**. P0.5(훅 보상)와 세트로 의미 생김 | 하 |
| 핀덱 클로즈업 컷 + 텍스트 팝 | CameraRig 상태 추가 + "STRIKE!/SPARE!" DOM 팝 + 네온 플래시 | 중 |
| 카메라 셰이크 | 임팩트 contact force 크기 비례, 짧게 | 하 |
| 사운드 — 굴림 럼블 | 현 SoundManager는 `playHit` 단일 합성음뿐. 럼블은 노이즈+필터 지속 루프 + 매 프레임 공 속도 피드 (연결 신설) | 중 |
| 사운드 — 크래시 구분 | contact 이벤트는 force 크기만 줌 → 짧은 윈도우로 충돌 클러스터 카운트해 1~2핀 딸각 vs 풀랙 크래시 구분 | 중 |
| 사운드 — 스트라이크 환호 | **합성으론 불가, 실음원 필요** → P4 에셋과 묶기. GameState에 strike/spare 이벤트 노출이 선행 (P2 전반의 공통 의존) | 중 |
| 핀세터 스윕바 | 데드우드 제거가 현재 순간이동 → 스윕바가 쓸어가는 애니로 가림. 물리 없음, 시각 전용 | 중 |

## P3. 숙련 깊이 — 물리를 게임플레이로 환전

공통 선행: `hookFactor`/`OIL_END_Z`가 모듈 전역 상수이고 **Lane·Ball·Controls 3곳이 직접 참조** → 런타임 변경하려면 파라미터화(오일 상태 객체) 선행. 이거 하나 하면 아래 둘 다 싸짐.

| 항목 | 구현 메모 | 난이도 |
|---|---|---|
| 예측선 난이도 | 현재 예측선은 오차 ~1cm 오라클이라 조준 스킬을 대체해버림. 이지=풀라인 / 노멀=오일 존까지 / 프로=레인 화살표만. **지향점 결정(상단) 종속** | 하 |
| 오일 패턴 프리셋 | 하우스샷/숏/롱. 상수는 준비됐으나 위 파라미터화 필요 — "공짜" 아님 | 하~중 |
| 레인 전환(오일 마름) | 프레임 경과로 드라이 존 전방 확장. 파라미터화 선행 시 "하". 예측선도 같은 오일 상태를 봐야 정합 유지 | 하~중 |
| 릴리스 타이밍 | 파워 게이지 핑퐁 정점 근처 릴리스=정확, 벗어나면 aim 노이즈. 실행 스킬 추가 (P0.5 레버 ②로 승격 검토 중 — P0.5 참조) | 하 |
| 터치 스핀 입력 | 모바일 결정(상단) 시: 릴리스 직전 좌우 플릭 or 스핀 슬라이더. 설계 도안 §8의 플릭 안 참조 | 중 |

## P4. 콘텐츠/리텐션 — 나중

- 볼 컬렉션: 훅 포텐셜·컨트롤 다른 공 언락 (BallSpec 확장)
- 스페어 드릴 챌린지: 특정 핀 배치 픽업 (7-10 스플릿 등) — PinSet에 임의 배치 함수
- 로컬 2인 교대전: ⚠️ "GameState 2개"가 아님 — 분리할 건 **점수 상태(`frame/ball/rolls`)뿐**, 물리 객체(PinSet/Ball)는 공유 + 턴 전환 시 리셋. Hud 2인 점수판 대응 (중)
- 데일리 시드 챌린지 (시드 고정 오일 패턴)
- 비주얼/사운드 에셋: GLTF 핀·공, HDRI, 실음원(환호 등 P2 의존분) — 게임성보단 프레젠테이션, 마지막
- 범위 밖 백로그(하이스코어 외 통계·접근성·그래픽 옵션)는 GAME_DESIGN.md §12 P2 섹션 참조
- 파울 라인 판정(도안 §7의 'F')은 1인 게임 임팩트 낮아 **의도적 범위 밖** — 필요해지면 발사 위치 개념부터 신설

## 추천 순서

**P0 → P0.5 → P1 → P1.5 → P2 → P3 → P4.**
P0 없이는 방향 결정 근거가 없고, P0.5 없이는 훅(핵심 자산)이 장식이고, P1 없이는 남에게 보여줄 수 없고, P1.5가 동기를 만들고, P2가 체감을 가장 크게 바꾸고, P3는 잘 만든 물리의 보람을 회수. P4는 리텐션 무관심 결정에 따라 추가 하향 — 스페어 드릴·볼 컬렉션 정도만 생존 후보.
(모바일 지원을 결정하면 "터치 스핀 입력"이 P1급으로 승격되는 것만 예외.)
