# 인게임 UI 개편 설계

> 작성: 2026-06-16 (9차 커밋 `9fcc1de` UI 전면 개편 *이후* 실기기 재검토)
> 베이스: iPhone 17 Pro 실기 스크린샷 3장(스핀 0 / L1.0 / R1.0) + 레퍼런스 조사 + 현 코드 정독
> 관련 코드: [Controls.ts](../src/input/Controls.ts) · [Hud.ts](../src/ui/Hud.ts) · [theme.ts](../src/ui/theme.ts)

## 한 줄 요약

`9fcc1de`의 UI 개편으로 **점멸 해소·조준선 곡선화·세로 파워게이지**까지 들어왔고 톤(네온 신스웨이브)은 일관됨. 그러나 실기기에서 보면 **① 빈 파워게이지(정보 0) · ② 스핀↔파워 입력 비대칭 · ③ 중립 조준선이 밝은 레인에 묻힘 · ④ 공이 스핀 패널에 가림** 4가지가 "한 번 보면 어색한" 문제로 남아 있다. 레퍼런스(PGA 2K25 EvoSwing / 8 Ball 가이드라인)가 한 방향을 가리킨다: **입력 UI는 가볍고 한 쌍으로, 정보는 더 또렷하게.**

> **구현 완료 (2026-06-16, 최종):** ① 파워 게이지 실체화 · ② 하단 도크 통합(스핀 좌하단 2단 / 파워 우하단 세로 · 시안 통일) · ③ 조준선 고대비 + 끝점 링 · ④ 공 가림 해소 + **조준선 거동 수렴**(파워와 분리해 *고정* — 차징에 안 흔들림 / `REF_Z=14` 곡선을 `DRAW_Z=5`로 곡률 보존 압축). tsc 클린 + 데스크톱 프리뷰 검증. **남은 것: 실기기(iPhone, coarse 분기) 최종 확인 + 미커밋.** 선택(점수판 축약) 미착수.
> 아래 "개편 전 상태 / 진단 / 개선 방향"은 **착수 시점의 베이스라인·계획 기록**이고, 무엇이 최종 적용됐는지는 [작업 체크리스트](#작업-우선순위-체크리스트)가 정본이다.

## 개편 전 상태 (베이스라인 — `9fcc1de` 시점)

> 아래 표·진단의 코드 라인/값(`Z_CAP=10`, `L###`, "스핀 풀폭", "빈 게이지" 등)은 **착수 시점 기준**이며 구현으로 바뀌었다 — 최종은 체크리스트 참조.

`9fcc1de` 시점 인게임 UI는 3개 레이어로 구성:

| 요소 | 위치 | 코드 | 비주얼 |
|---|---|---|---|
| 점수판 + 상태줄 | 상단 중앙 | `Hud.renderSheet` / `Hud.status` | `applyPanel` 글래스, 현재 프레임 골드 펄스 |
| 조준 곡선선 | 레인 위 3D | `Controls.updateAimArrow` (`Line2` 외곽선+코어 2겹) | L=시안/R=앰버/0=흰색, 끝 페이드, `Z_CAP=10` |
| 스핀 바 | 하단 풀폭 가로 | `Controls.spinWrap`/`spinTrack`/`spinThumb` | 중앙 0, 좌(시안)/우(앰버) 채움 + 화살표 수치 |
| 파워 게이지 | 우측 하단 세로 | `Controls.powerWrap`/`gaugeTrack`/`gaugeFill` | 빈 트랙 + 차징 시 초록→빨강 채움 |

**공통 토큰은 이미 있음** — `theme.ts`의 `applyPanel()`(딥퍼플 그라데+네온 보더+블러), `NEON` 팔레트, `ensureNeonStyles()`(`.neon-range`). 즉 디자인 시스템의 *뼈대*는 있고, 문제는 그 위의 **적용 방식**이다.

## 진단 (우선순위 순)

### ① 파워 게이지가 "정보 0" — 빈 캡슐로 읽힘 ⭐
- **증상**: 스크린샷 3장 모두 우측 세로 바가 비어 있음. 라벨·눈금·휴지 채움이 전혀 없어( `Controls.ts` L136에서 POWER 라벨을 폭 문제로 제거) 처음 보면 "이게 파워인지" 모름.
- **코드**: `gaugeTrack` 배경 `rgba(255,255,255,0.1)` + `gaugeFill` 차징 시에만 `height` 증가. 평상시 affordance 0.
- **레퍼런스 갭**: PGA 2K25 EvoSwing도 우측 세로 미터지만 **명확한 채움 + 타깃 존 마커 + 눈금**이 있어 "얼마 찼고 어디가 목표인지" 보임. 구조는 같고 정보량만 정반대.

### ② 스핀 ↔ 파워 입력 비대칭
- **증상**: 같은 "투구 입력"인데 스핀은 **하단 풀폭 가로 바**, 파워는 **우측 짧은 세로 바**. 형태·위치·크기가 모두 달라 한 쌍으로 안 읽힘.
- **코드**: `spinWrap`(`width:440px` 데스크톱 / 풀폭 터치, `bottom:10px`) vs `powerWrap`(`width:14px`, `height:180px/26vh`, `right`). 액센트 색도 스핀=`NEON.purple`, 파워=`NEON.cyan`, 상태줄=`cyan`, 시트=`gold/cyan`로 **의미 없이 흩어짐**.
- **참고(코드 주석 드리프트)**: `Controls.ts` L118 주석은 "스핀=좌하단·파워=우하단"이라 적혀 있으나 실제 스핀은 풀폭 가로. 개편 시 주석도 정리.

### ③ 중립(흰색) 조준선이 밝은 레인에 묻힘
- **증상**: 스핀 0(1번 샷)에서 흰 선이 거의 안 보임. 시안/앰버(2·3번)는 그나마 보임 → 가장 자주 쓰는 중립이 가독성 최악.
- **코드**: `updateAimArrow` L472 페이드 `Math.pow(i/last,1.1)*1.15` → 중반쯤 레인색(tan `0xcdb892`)에 완전히 녹음 + `Z_CAP=10`로 짧음. 흰색→tan lerp는 밝은 바닥에서 대비를 빨리 잃음.
- **레퍼런스 갭**: 8 Ball Pool류는 밝은 배경에서도 또렷한 **고대비 궤적선 + 끝점 접촉 인디케이터**가 표준. 우리는 끝이 사라지기만 하고 "어디 떨어지나" 표시가 없음.

### ④ 공이 스핀 패널에 가림
- **증상**: 3장 모두 파란 공이 하단 스핀 패널 뒤로 반쯤 잘림. `9fcc1de`에서 "단일 줄로 얇게"(`Controls.ts` L165) 시도했으나 풀폭이라 중앙에서 여전히 겹침.
- **코드**: `spinWrap` 풀폭 + `bottom:10px`, 공은 `BALL_START_Z`에서 화면 하단 중앙 렌더 → 패널과 공이 같은 영역 점유.

## 개선 방향 (제안)

레퍼런스가 공통으로 가리키는 원칙: **"입력 UI를 가볍고 일관되게, 정보는 더 또렷하게."**

### 1순위 — 파워 게이지 실체화 (①)
- 평상시에도 보이는 휴지 트랙 + 차징 채움(현 그라디언트 유지).
- **파워 존 마커**: 스트라이크 최적 파워 구간(직구/훅별 다름 — `MAX_SPEED` 매핑 기준)을 트랙에 띠/눈금으로 표시.
- 작은 라벨 or 아이콘(🎳/⚡) 1개 — 폭 안 넘게 위/아래 배치.
- 터치: 핑퐁 차징이라 "지금 값"이 계속 움직임 → 현재 위치 마커를 또렷하게.

### 2순위 — 스핀·파워 한 쌍으로 정렬 + 액센트 의미화 (②)
- 옵션 A: **둘 다 하단 도크**로 묶되 좌(스핀)/우(파워) 대칭, 가운데 비워 공·조준선 노출.
- 옵션 B: **양쪽 세로 대칭**(좌 스핀 세로 / 우 파워 세로) — 엄지 도달성↑(모바일).
- 액센트 색 규칙 통일: 예) 입력=시안 계열, 상태/현재=골드, 경고=레드. 지금처럼 purple/cyan/gold 난립 정리.

### 3순위 — 조준선 가독성 (③)
- 중립색을 순백 대신 **외곽 다크 케이스 대비↑ + 코어 글로우↑**로(밝은 레인에서 살아남게). 페이드 시작점을 뒤로(`L472` 지수/배율 완화), `Z_CAP` 소폭↑ 검토.
- 끝에 **임팩트/착지 포인트 마커**(작은 링/점) — "여기로 간다"를 명시. 8 Ball 패턴.

### 4순위 — 공 가림 해소 (④)
- 스핀 도크를 중앙에서 비키게(좌측/양쪽) 하거나, 패널 배경을 트랙만 남기고 더 투명하게.
- 또는 공 스폰 화면 위치를 살짝 위로(카메라 `CameraRig` AIMING 구도) — 단 ②와 함께 봐야 함.

## 작업 우선순위 체크리스트

- [x] **(P1)** 파워 게이지 실체화 — ⚡ 아이콘 + 흐린 최적존 띠(0.6~0.9) + 진입 경계선 (`Controls.ts`, tsc 클린 + 프리뷰 확인). 실기기 재확인만 남음.
- [x] **(P2)** 하단 도크 통합 — 스핀 좌하단 2단(라벨+값/트랙)·파워 우하단 세로 같은 베이스라인, 액센트 시안 통일·중립 썸 ice (`Controls.ts`). tsc 클린 + 데스크톱 프리뷰 확인. (※ `theme.ts` 시맨틱 토큰 신설 대신 기존 `NEON.cyan/ice` 재사용으로 통일.)
- [x] **(P3)** 조준선 중립 고대비화 + 끝점 마커 — 페이드 `pow(i/last,1.1)*1.15`→`min(0.82,pow,2.0)`(중반에 녹던 것 해소) + 경로 끝 스핀색 타깃 링(`RingGeometry`, z=Z_CAP=10, 포켓 결과는 계속 숨김). tsc 클린 + 프리뷰 확인(중립 흰선·시안/앰버 곡선·링).
- [x] **(P3 후속)** 차징 중 조준 화살표 거동 — **3차 수렴**. ① 끝점 `Z_CAP` 하드클램프(끝점 버벅은 남고 "맥시멈 고정" 느낌) → ② 길이를 파워 비례로 자라나게(버벅은 사라졌으나 차징이 핑퐁 0↔1이라 길이가 쉴 새 없이 펌프질 = 부자연, 실기기 확인) → ③ **조준선을 파워와 완전 분리해 고정**(대표 파워 `p=0.6`·`endZ=Z_CAP` 상수). 차징해도 조준선은 멈춰 있고 파워는 우측 게이지가 전담 = 당구식 안정 가이드. 끝점 트림 유지(매끄러운 끝, 버벅 없음). 조준(aim)·스핀 입력엔 그대로 반응. tsc 클린 + 프리뷰 검증(게이지 풀/저 두 위상에서 조준선 길이 불변 확인). **교훈: 핑퐁 차징에 라인 길이/모양을 묶으면 무조건 펌프질 → 입력 가이드는 파워와 분리.**
- [x] **(P3 후속2)** 조준선 길이·곡률 — 너무 길어(z=10, 미드레인) 줄이되, 그냥 `DRAW_Z`까지만 적분하면 그 구간이 오일존(직진)이라 곡률이 안 보였음. → **`REF_Z=14` 곡선을 적분해 모양 확보 후 시작점 기준 균일 스케일 `k=(DRAW_Z−start)/(REF_Z−start)`로 비례 축소**(`DRAW_Z=5`). `REF_Z`↑ = 곡률↑(드라이존 훅 더 포함), `DRAW_Z` = 그리는 길이. 짧아도 훅 곡률 보존(축소판 바나나), 균일 스케일이라 조준 방향(각도) 불변. tsc 클린 + 프리뷰(스핀 L 곡선 확인).
- [x] **(P4)** 공 가림 해소 — P2 도크가 가운데를 비워 공·조준선 밑동 노출(데스크톱 프리뷰 확인). 실기기(coarse 46vw) 재확인만 남음.
- [x] **점수판(Hud) 개선** — **방향 A "한 줄 꽉 채우기" 채택**(2행·가로 스크롤 둘 다 사용자가 기각). 솔로/멀티 활성 = 10프레임을 `row{width:min(96vw,자연폭)}` + 프레임/셀 `flex-basis:0` 비례 분배(스크롤 0), 멀티 비활성 = `renderMini`(F# + 현재 프레임 마크 + 큰 총점). `overflowX:auto` 제거. 누적 폰트 `SCORE_FS`는 좁은 셀(프레임 ~26px)에 3자리 여유 위해 기본 12/14 유지(+1 시도→3자리 90%로 답답해 원복). **버그·수정**: 1차에 `fit-content`+`flex:0 1 CELLpx`로 했더니 iPhone 실기에서 **초반 빈 칸이 폭 0으로 collapse**(점수판이 작은 캡슐로 찌그러짐) — `fit-content`가 빈 내용을 0으로 산정. **정해진 폭 + `flex:1 1 0`(basis 0, 내용 무관 분배)로 교정.** (1차 격리 측정이 "채운 점수표"라 빈칸 케이스를 놓침 → 빈 상태로 재검증.) tsc 클린 + vitest 22/22 + 프리뷰 격리 재측정(빈칸·채운칸 / 320·393px 전부 한 줄·넘침 0; 솔로 셀 11.8~13px, 멀티 활성 7.3~11px). **남은 것: iPhone 실기 재확인.** (아래 "점수판(Hud) 개선" 섹션이 정본.)
- [x] 코드 주석 드리프트 정리 (`Controls.ts` 하단 도크 주석 — P2에서 갱신)

## 검증 방법

- `npx tsc --noEmit` + `npx vitest run`(현재 22/22) — 회귀 없음 확인.
- **실기기 필수** — 헤드리스 프리뷰는 메뉴 오버레이가 씬을 가리고 rAF가 멈춰 인게임 UI 관측 불가(점멸 조사 때와 동일 제약, [PROGRESS.md](./PROGRESS.md) 참조). iPhone 스크린샷으로 4문제 해소 재확인.
- 입력 회귀 주의: 스핀 바 `pointerdown`은 캔버스 차징과 독립(`draggingSpin`) — 레이아웃 바꿔도 멀티터치 오발사/`pointercancel` 고착 가드 유지.

## 결정 완료 (2026-06-16)

1. **②의 레이아웃** → **하단 도크 통합** — 스핀(가로)+파워를 하단 한 묶음으로, 같은 패널 스타일·액센트 규칙으로 통일. 스핀은 가로 유지(좌우=훅 방향 직관성 보존), 가운데를 비워 공·조준선 노출.

## 점수판(Hud) 개선 — 구현 완료 (방향 A, 11차)

핵심 코드: `Hud.renderSheet`(풀 시트) · `Hud.renderMini`(멀티 비활성) · `CELL`(coarse 13px, 이제 flex-basis) · `row`(`fit-content`+`maxWidth:96vw`).

**원칙: 스크롤 0 — 항상 한눈에(HUD는 글랜스용).** 가로 스크롤(`overflowX:auto`)은 발견성↓·제스처 혼동이라 **제거**, "다 들어가게(fit)"로 대체.

### 최종 결정: 방향 A — 한 줄 꽉 채우기
2행(5+5)과 가로 스크롤 **둘 다 기각**(사용자). 대신:
- **솔로 / 멀티 활성**: 행에 **정해진 폭** `row { width:min(96vw, NAT_SHEET(+NAME_W)); max-width:96vw }`을 주고, 프레임 `flex:{2|3} 1 0`·셀 `flex:1 1 0`(basis 0)로 **비례 분배**. 여유 있으면 자연폭, 좁으면 균일 축소. 2칸/마지막 3칸 = grow 2:3이라 **모든 셀 폭 동일**. 정석 "한 줄 10프레임"(중계 그래픽) 관례 유지 = 2행 기각 사유 해소. ⚠️ `fit-content`는 금물 — 초반 빈 칸을 0으로 산정해 collapse(실기 확인).
- **멀티 비활성(AI)**: `renderMini` — `F{프레임}` + 현재 프레임 마크 + **큰 총점**(확정분, `frameScores` 마지막 정의값). 폭·높이 절약, 상대는 총점이 핵심이라 손실 적음. **게임 종료 시엔 전원 풀 시트**로 최종 비교.
- **공통 폴리시**: 누적 총점 폰트는 기본 12/14 유지(좁은 셀이라 3자리 여유 우선 — +1은 답답해 원복), 현재 프레임 골드 펄스 유지.
- **유지(이미 옳음)**: 보너스 확정 전 누적 숨김(`frameScores` undefined 게이팅 — 점수판 #1 베스트프랙티스), 표준 마크 X/–.

### 검증
tsc 클린 + vitest 22/22 + **프리뷰 격리 측정**(인게임 HUD는 메뉴 오버레이로 직접 관측 불가 → 동일 CSS 구조 주입해 측정). ⚠️ **1차 측정은 "채운 점수표"로 해서 빈칸 collapse를 못 잡음 → iPhone 실기에서 발견(찌그러진 캡슐).** `fit-content`→`min(96vw,자연폭)` + `flex:1 1 0`로 교정 후 **빈칸 상태로 재검증**: 320·393px·솔로·멀티 **전부 한 줄·넘침 0**. 솔로 셀 11.8~13px, 멀티 활성 7.3(320)~11px(이름 패널이 폭 잠식), 미니 행은 항상 여유. **교훈: fit-to-width 레이아웃은 빈/초반 상태로 검증할 것(채운 상태가 collapse를 가린다).** **남은 것: iPhone 실기 재확인 후 커밋.**

**트레이드오프 기록**: A는 좁은 폰(320, 구형 SE)에서 멀티 활성 셀이 7.6px까지 작아지는 걸 감수(스크롤·2행 회피 우선). 일반 폰(375+)은 11px+로 무난. fit-to-width가 desktop은 그대로(자연폭 < 96vw), 좁은 폰만 축소 → 단일 메커니즘으로 솔로·멀티·데스크톱 동시 처리.

**레퍼런스**: [thesportofbowling 점수표](https://www.thesportofbowling.com/blog/bowling-score-sheet/) · [templatelab(누적 타이밍 실수)](https://templatelab.com/bowling-score-sheet/) · [Bowling Scoreboard 앱](https://apps.apple.com/us/app/bowling-scoreboard/id313837528).
